# WordMiner 项目复盘

> 首次复盘：2026-07-07。用 `project-retro` skill 模板产出。

## 1. 为什么做？用户痛点是什么？

看 YouTube 学英语时的核心矛盾：**视频里的生词转瞬即逝**。听到不认识的词想查，就得暂停 → 手动敲进词典 → 记住它出现在哪句话里 → 回来继续看。一个视频打断几十次，学习动力被磨光。

痛点本质：**"发现生词 → 查义 → 带语境记忆"这条链路手动做太重。**

## 2. 用户是谁？

- **主要用户：作者自己**——有一定英语基础、想用真实材料提升词汇的自学者。
- **潜在用户：中级英语学习者**——能大致看懂 YouTube，但被长尾生词卡住。

> 诚实说：这一版是先满足自己的工具，还没做真正的用户调研。

## 3. 为什么不用传统方法 / 现成方案？

- **不直接看字幕**：字幕能懂大意，但不帮你挑出值得学的生词、不给中文释义/音标、不留语境卡片。
- **不用 ChatGPT**：能用，但每次要自己找字幕→复制→写 prompt，结果不结构化、无法沉淀成可复习卡片。它是通用工具，不是这个场景的顺手工具。
- **不用现成插件（Language Reactor / Trancy）**：其实**能用且更成熟**（复盘中途才查到）。自己做更多是练手 + 完全可控 + 未来能加自定义 AI 功能。**如果纯为效率，装 Trancy 就够了——这点要对自己诚实。**

## 4. AI / 核心技术在整个流程中的作用？

真实流程（**当前版本几乎没用 LLM**）：

```
输入视频链接
    ↓
提取字幕            ← Supadata API（托管抓取服务，非 AI）
    ↓
挖词（选高价值生词） ← 本地规则算法：停用词过滤 + 词长/词频打分（非 AI）
    ↓
单词解释            ← 谷歌翻译接口(中文) + dictionaryapi.dev(音标/词性/英文释义)
    ↓
（AI 总结 / 生成练习：未实现，属路线图）
```

**关键澄清**：名字带"智能"，但挖词是规则算法、释义是词典 API，**没有 LLM 层**。"AI 总结、生成练习"是未来路线图，不是现状。

## 5. 技术架构

```
前端 public/index.html（原生 JS，无框架）
    ↓ 同源 API
Node.js server.js（无框架，纯 http 模块）
    ├── /api/subtitles → Supadata API（抓字幕，绕过 IP 封锁）
    ├── /api/lookup    → 谷歌翻译 + dictionaryapi.dev（查词）
    └── /api/translate → 谷歌翻译（整句）
    ↓ 部署
Render（免费套餐，push main 自动部署）
```

真实架构没有"LLM API"这一层。

## 6. MVP 是什么？

第一版**只有**：粘贴 YouTube 链接 → 抓英文字幕 → 算法挖出 top 18 生词 → 每张卡片显示"生词 + 语境句 + 中文 + 音标 + 英文释义"。

**没有**：登录、生词本、复习、AI 总结、练习题、多语言。

## 7. 遇到什么问题？

1. **核心拦路虎：YouTube 封锁云服务器 IP**。本地（住宅 IP）抓字幕正常，一部署到 Render（机房 IP）就全部 `LOGIN_REQUIRED`。
2. **cookie 陷阱**：以为登录 cookie 能绕过，实测四种会话模式全失败。**耗时最多的坑。**
3. **字幕下载空**：带 cookie 能登录但 POT token 丢失，字幕 0 字节；本地会话又和 cookie 冲突。
4. **一堆小坑**：cookie 粘贴带换行→"invalid header value"；Render 免费实例内存小把 4合1 诊断请求跑崩；冷启动 30 秒。

## 8. 怎么解决？

**真正的解法是想清楚了问题的性质，而不是"用了某工具"：**

- 关键转折是**承认"抓字幕"在机房 IP 上自己做不到**，停止和 youtubei.js / cookie 死磕。
- 中途试过一个能跑但笨的方案：**本机当字幕代理**（Render 前端 + 本机 helper 抓字幕）。能通但要开代理、电脑常开、仅 Chrome——体验太差，废弃。
- 最终解法来自**用户提醒"别从零造火箭，找现成工具"**。搜索发现业界对"云 IP 被封"早有标准答案：**托管字幕 API**。换成 **Supadata**（它自己维护代理池），Render 后端一个 HTTP 请求就拿到字幕，所有约束一次性消失。

> 一句话：**问题不是"怎么让我的爬虫绕过封锁"，而是"这件事该不该我自己做"。答案是外包给专门的服务。**

## 9. 如果重新做一次 / 学到了什么？

**重做会怎么优化：**
1. **先调研再动手**。第一天就搜"YouTube transcript API 2026"，直接发现 Supadata，跳过 cookie/helper 两条弯路。
2. **先验证最难的环节**。抓字幕是命门，应在**部署环境（机房 IP）第一个验证**，别被本地住宅 IP 掩盖真问题。
3. **别过早优化会被废弃的方案**。本机 helper 的 UI/bat 写完即弃，纯浪费。
4. **想清楚 AI 到底用在哪**，别名不副实。

**学到了什么（可迁移原则）：**
1. **"不要从零造火箭"最值钱**——遇到技术难题先问"是不是已经有人专门解决了"。
2. **区分"核心价值"与"基础设施"**——抓字幕是基础设施（外包），挖词逻辑+学习体验才是核心（自己做）。
3. **在真实环境验证假设**——"本地能跑"≠"线上能跑"，尤其涉及 IP/网络/权限。
4. **诚实评估要不要做**——现成方案更成熟时，承认自己做的真实理由是练手/可控/定制。

## 下一步（投入产出比最高的增强）

现在真正缺 **AI 那一层**，也是区别于 Trancy 的机会点：
- **用 LLM 替代规则挖词**：按"对学习者的价值"挑词，而非词频。
- **AI 语境化释义**（推荐先做）：解释词在**这句话里**的具体含义，而非词典通用义。
- **生成练习**：基于挖出的词自动出填空/造句题。
